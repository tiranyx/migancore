## 1. FastAPI JWT RS256 — Authentication Architecture

### Key Findings

**Package Choice (2024-2025 Standard)**
- `pyjwt[crypto]` for JWT signing/verification (NOT python-jose — less maintained)
- `pwdlib[argon2]` for password hashing (NOT bcrypt — Argon2 is OWASP 2024 recommendation)
- `fastapi.security.HTTPBearer` + `OAuth2PasswordBearer` for token transport

**RS256 Key Management**
```python
# Pattern: Load private key from file, verify with public key
from jwt import PyJWKClient

jwks_client = PyJWKClient(JWKS_URI)  # or load local PEM
signing_key = jwks_client.get_signing_key_from_jwt(token)
payload = jwt.decode(token, signing_key.key, algorithms=['RS256'])
```
- Our setup already has RSA keys at `/etc/ado/keys/` (generated by day1-setup.sh)
- Private key for signing (API only), public key for verification (any service)

**Multi-Tenant JWT Claims Structure**
```json
{
  "sub": "user-uuid",
  "org_id": "tenant-uuid",
  "role": "admin",
  "scope": "ado:read ado:write agent:create",
  "iat": 1714700000,
  "exp": 1714703600,
  "aud": "api.migancore.com"
}
```
- `org_id` = tenant isolation key (RLS filter in Postgres)
- `scope` = RBAC permission strings (space-separated)
- `aud` = audience validation prevents token reuse across services

**Token Lifecycle Best Practices**
| Token Type | Expiry | Storage | Usage |
|------------|--------|---------|-------|
| Access Token | 15-30 min | Memory only | API requests |
| Refresh Token | 7-30 days | httpOnly cookie | Token refresh endpoint |
| API Key | No expiry | Secure vault | Service-to-service |

**Token Revocation**
- Redis-backed blacklist for revoked tokens (before expiry)
- Key pattern: `jwt:blacklist:{jti}` with TTL = remaining token lifetime
- `jti` (JWT ID) claim required for revocation support

**FastAPI Dependency Pattern**
```python
from fastapi import Depends, HTTPException
from fastapi.security import HTTPBearer

security = HTTPBearer()

async def get_current_user(credentials: HTTPAuthorizationCredentials = Depends(security)):
    token = credentials.credentials
    payload = verify_jwt(token)
    user = await get_user_from_db(payload["sub"])
    if not user or user.org_id != payload.get("org_id"):
        raise HTTPException(403, "Tenant mismatch")
    return user
```

## 2. LangGraph Director — State Machine Design

### Key Findings

**Core Concepts**
- **Nodes**: Python functions that process state (LLM calls, tool calls, custom logic)
- **Edges**: Transitions between nodes (conditional edges = routing logic)
- **State**: Shared TypedDict passed to every node — single source of truth
- **Cycles**: First-class loops enable iterative refinement (plan → execute → evaluate → revise)

**State Schema Pattern for MiganCore**
```python
from typing import TypedDict, List, Optional
from langgraph.graph import StateGraph, END

class ADOState(TypedDict):
    # Input
    user_query: str
    tenant_id: str
    user_id: str
    
    # Routing
    intent: Optional[str]  # classified by Director
    confidence: Optional[float]
    
    # Agent outputs
    plan: List[str]
    research_results: List[dict]
    draft_response: Optional[str]
    critique: Optional[str]
    
    # Control
    iteration_count: int
    max_iterations: int
    needs_human: bool
    error: Optional[str]
```

**Director Node Pattern**
```python
def director_node(state: ADOState) -> ADOState:
    """Classifies intent and routes to appropriate specialist."""
    # LLM call with system prompt: "You are the Director. Classify intent..."
    classification = llm.classify_intent(state["user_query"])
    state["intent"] = classification.intent
    state["confidence"] = classification.confidence
    return state

def route_by_intent(state: ADOState) -> str:
    """Conditional edge: returns next node name."""
    if state["confidence"] < 0.7:
        return "clarify"
    routing_map = {
        "code": "code_agent",
        "research": "research_agent", 
        "creative": "creative_agent",
        "analytics": "analytics_agent",
    }
    return routing_map.get(state["intent"], "general_agent")

# Build graph
builder = StateGraph(ADOState)
builder.add_node("director", director_node)
builder.add_node("code_agent", code_agent_node)
builder.add_node("research_agent", research_agent_node)
builder.add_node("clarify", clarify_node)

builder.set_entry_point("director")
builder.add_conditional_edges("director", route_by_intent)
builder.add_edge("code_agent", END)
builder.add_edge("research_agent", END)
# ... etc

graph = builder.compile()
```

**Production Checkpointer: PostgresSaver**
- `MemorySaver` = in-process only, lost on restart ❌
- `PostgresSaver` = persists to PostgreSQL, survives restart ✅
- Required for: `interrupt()`, multi-request workflows, crash recovery
- Our postgres container already has `ado` database — add `langgraph_checkpoints` table

```python
from langgraph.checkpoint.postgres import PostgresSaver

checkpointer = PostgresSaver(conn_string=os.getenv("DATABASE_URL"))
graph = builder.compile(checkpointer=checkpointer)

# Human-in-the-loop
result = graph.invoke(
    {"user_query": "Deploy production"},
    config={"configurable": {"thread_id": "session-123"}}
)
# If interrupt() fired, result contains interrupt value
# Resume with: graph.invoke(Command(resume="approved"), config=...)
```

**Parallel Fan-Out Pattern**
```python
# Multiple edges from one node = parallel execution
builder.add_edge("director", "research_agent")
builder.add_edge("director", "code_agent")
# Both agents run concurrently, results merged in state
# Use Annotated[list[T], operator.add] for list-append semantics
```

**Key Decision: LangGraph vs CrewAI**
| Factor | LangGraph | CrewAI |
|--------|-----------|--------|
| Control | Explicit graph | Natural language roles |
| State | TypedDict, checkpointed | Event-driven |
| Learning Curve | Steep | Gentle |
| Production Maturity | High (PostgresSaver, interrupt) | Medium (needs Redis bolt-on) |
| Our Use Case | ✅ Complex reasoning loops | ❌ Too abstract for our needs |

**Conclusion**: LangGraph is the right choice for MiganCore's "self-evolving" requirement. The explicit state machine maps cleanly to our ADO metaphor.

## 3. Letta 0.6.0 — Integration Patterns

**Package Name Changed**

- New: `pip install letta-client` + `from letta_client import Letta`
- ⚠️ Our docker-compose uses `letta/letta:0.6.0` image — SDK must match

**Agent Lifecycle**
```python
from letta_client import Letta
import os

client = Letta(
    base_url="http://letta:8283",  # Internal Docker network
    api_key=os.getenv("LETTA_PASSWORD")  # Or auth token

agent = client.agents.create(
    name="ado-memory-core",
    model="openai/qwen2.5:7b-instruct",  # Or local Ollama model
    memory_blocks=[
        {"label": "persona", "value": "I am the MiganCore ADO..."},
        {"label": "human", "value": "Fahmi Wol is my creator..."},
        {"label": "context", "value": "Project: MiganCore v0.1.0..."}
    ],
    tools=["web_search", "fetch_webpage", "code_interpreter"],
    system="You are the persistent memory core...",
    tags=["ado", "core", "persistent"]
)

# 2. Send message
response = client.agents.messages.create(
    agent_id=agent.id,
    input="What did we discuss about LangGraph yesterday?",
    max_steps=10
)

# 3. Stream response
stream = client.agents.messages.create(
    agent_id=agent.id,
    input="Explain the Director pattern",
    streaming=True,
    stream_tokens=True

```

**Integration Architecture for MiganCore**
```
┌─────────────┐     ┌─────────────┐     ┌─────────────┐
│   FastAPI   │────▶│  LangGraph  │────▶│   Letta     │
│   Gateway   │     │  Director   │     │  Memory     │
└─────────────┘     └─────────────┘     └─────────────┘
       │                   │                   │
       ▼                   ▼                   ▼
  JWT Auth            State Machine       Persistent
  RBAC                Routing Logic       Memory Blocks
```

**Key Design Decision: Letta as Memory Layer, NOT primary LLM**
- Letta handles: persistent memory, conversation history, agent state
- Ollama handles: inference, code generation, reasoning
- LangGraph Director handles: routing, workflow orchestration
- This separation avoids vendor lock-in and keeps inference local

**Container Port Verification Needed**
- Our docker-compose: Letta exposes `8083/tcp` internally
- But Letta service typically runs on port `8283` or `8083`
- **Action**: Verify actual port when container starts, adjust `base_url`

## 4. SimPO + QLoRA — Training Pipeline

### Key Findings

**Training Pipeline: 3 Stages**
```
Stage 1: SFT (Supervised Fine-Tuning)
   ├── Dataset: Instruction-following pairs
   ├── Method: QLoRA (4-bit quantization + LoRA adapters)
   ├── Output: Base instruction-tuned model
   └── Tools: transformers Trainer, PEFT, BitsAndBytes

Stage 2: Preference Alignment (SimPO)
   ├── Dataset: Paired preferences (chosen vs rejected)
   ├── Method: SimPO (Simple Preference Optimization)
   ├── Key difference from DPO: No reference model needed
   └── Tools: TRL library (transformers reinforcement learning)

Stage 3: Merge & Deploy
   ├── Merge LoRA adapters into base model
   ├── Quantize to GGUF for Ollama
   └── Push to Ollama model registry
```

**QLoRA Configuration (Verified Working Config)**
```python
# BitsAndBytes 4-bit quantization
bnb_config = BitsAndBytesConfig(
    load_in_4bit=True,
    bnb_4bit_quant_type="nf4",
    bnb_4bit_compute_dtype=torch.float16,
    bnb_4bit_use_double_quant=True,
)

# LoRA configuration
lora_config = LoraConfig(
    r=16,              # Rank
    lora_alpha=32,     # Alpha (typically 2*r)
    target_modules=["q_proj", "v_proj", "k_proj", "o_proj"],
    lora_dropout=0.05,
    bias="none",
    task_type="CAUSAL_LM",
)
```

**SimPO Hyperparameters (from Princeton NLP GitHub)**
| Parameter | Qwen2.5-7B-Instruct Recommendation |
|-----------|-----------------------------------|
| learning_rate | 5e-7 (reasoning) to 1e-6 (general) |
| beta | 2.0 to 10.0 (SimPO needs larger than DPO) |
| gamma_beta_ratio | 0.5 (grid search 0-1) |
| batch_size | 128 total (with grad accumulation) |
| epochs | 1-3 |
| max_length | 2048-4096 |

**Key Advantage of SimPO over DPO**
- SimPO does NOT need a reference model (saves ~50% VRAM)
- Simpler objective: maximize margin between chosen and rejected
- Better for smaller models (7B) with limited compute

**VPS Training Reality Check**
- Our VPS: 32GB RAM, no GPU
- QLoRA 7B in 4-bit: ~5GB VRAM needed → Fits in RAM with CPU offloading
- Training time on CPU: ~10-20x slower than GPU
- **Recommendation**: Use `device_map="auto"` with `low_cpu_mem_usage=True`
- Alternative: Train on Google Colab (free T4), export GGUF, import to VPS

**Data Format for SimPO**
```json
{
  "prompt": "Explain the concept of...",
  "chosen": "Clear, accurate explanation...",
  "rejected": "Vague, incorrect explanation..."
}
```
- Dataset: UltraFeedback (60k pairs, open source)
- For MiganCore: We need custom preference data (agent interactions rated by humans)

## 5. Celery + Redis — Task Queue Best Practices

### Key Findings

**Production Configuration**
```python
# celery_config.py
from celery import Celery
from kombu import Queue

app = Celery("ado")
app.conf.update(
    broker_url=os.getenv("REDIS_URL"),           # redis://redis:6379/0
    result_backend=os.getenv("REDIS_URL"),       # redis://redis:6379/0
    
    # Serialization
    task_serializer="json",
    accept_content=["json"],
    result_serializer="json",
    
    # Reliability
    task_acks_late=True,                    # Ack AFTER completion
    task_reject_on_worker_lost=True,        # Requeue if worker dies
    worker_prefetch_multiplier=1,           # One task at a time
    
    # Memory
    worker_max_tasks_per_child=1000,        # Restart worker after N tasks
    worker_max_memory_per_child=200000,     # 200MB max per child
    
    # Timeouts
    task_soft_time_limit=300,               # 5 min warning
    task_time_limit=600,                    # 10 min hard kill
    
    # Results
    result_expires=86400,                   # 24h result retention
    
    # Queues
    task_queues=(
        Queue("high", routing_key="high"),
        Queue("default", routing_key="default"),
        Queue("low", routing_key="low"),
    ),
    task_default_queue="default",
)
```

**Task Definition Pattern**
```python
from celery import shared_task
from celery.exceptions import MaxRetriesExceededError

@shared_task(bind=True, max_retries=3, default_retry_delay=60)
def process_agent_request(self, request_id: str, query: str, tenant_id: str):
    """Process an agent request with retry logic."""
    try:
        result = run_langgraph_workflow(query, tenant_id)
        return {"status": "success", "result": result}
    except TransientError as e:
        # Retry with exponential backoff
        raise self.retry(exc=e, countdown=60 * (2 ** self.request.retries))
    except PermanentError as e:
        # Dead letter — no retry
        return {"status": "failed", "error": str(e)}
```

**FastAPI Integration (Async-Safe)**
```python
from fastapi import FastAPI, BackgroundTasks

@app.post("/agent/ask")
async def ask_agent(request: AgentRequest):
    # Fire and forget — return task ID immediately
    task = process_agent_request.delay(
        request_id=str(uuid4()),
        query=request.query,
        tenant_id=request.tenant_id
    )
    return {"task_id": task.id, "status": "queued"}

@app.get("/agent/status/{task_id}")
async def get_task_status(task_id: str):
    result = celery_app.AsyncResult(task_id)
    return {
        "task_id": task_id,
        "status": result.state,  # PENDING, STARTED, SUCCESS, FAILURE
        "result": result.result if result.ready() else None,
    }
```

**⚠️ CRITICAL: Never call `.get()` inside async endpoint**
```python
# WRONG — blocks event loop
result = task.get(timeout=10)

# RIGHT — return task_id, poll separately
return {"task_id": task.id}
```

**Queue Strategy for MiganCore**
| Queue | Tasks | Priority | Max Runtime |
|-------|-------|----------|-------------|
| `high` | Auth, health checks, user-facing | Immediate | 5s |
| `default` | Agent inference, API calls | Normal | 5min |
| `low` | Training jobs, batch processing, backups | Background | 30min |

**Flower Monitoring**
- `pip install flower`
- `celery -A ado.celery_app flower --port=5555`
- Protect with `--basic_auth=user:password` or nginx auth
- REST API available for programmatic monitoring

## 6. Cross-Cutting Concerns

### Memory Budget (Updated with Research)
| Service | Idle | Loaded | Notes |
|---------|------|--------|-------|
| Existing VPS | ~5GB | ~5GB | SIDIX + Ixonomic + aaPanel |
| Postgres | 200MB | 500MB | pgvector extension loaded |
| Redis | 50MB | 100MB | With queue data |
| Qdrant | 100MB | 200MB | Vector index |
| Letta | 200MB | 500MB | Agent state |
| FastAPI | 100MB | 200MB | 2 workers |
| Celery Workers (x3) | 300MB | 600MB | 3 queues × 200MB |
| Ollama (qwen2.5:7b) | 0MB | 4.7GB | Lazy load, 10m keepalive |
| LangGraph Runtime | 100MB | 300MB | State machine overhead |
| **Total MiganCore** | **~1GB** | **~7GB** | |
| **Grand Total** | **~6GB** | **~12GB** | **Safe with 32GB + 8GB swap** |

### Security Considerations
1. **JWT Secret Rotation**: RS256 keys can be rotated without breaking active sessions (public key distributed, private key changed)
2. **Rate Limiting**: Implement per-tenant rate limits using Redis (`redis-rate-limits`)
3. **Input Validation**: All agent inputs must pass Pydantic validation before reaching LangGraph
4. **Sandboxing**: Agent code execution requires sandbox (gVisor or restricted Docker)

### Observability Stack
```
Langfuse (tracing)     → Track agent reasoning steps, token usage
Flower (Celery)        → Queue depth, worker health
Prometheus + Grafana   → Custom metrics (future)
FastAPI /health        → Liveness probe
FastAPI /ready         → Downstream dependency checks
```
