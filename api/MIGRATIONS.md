# MiganCore Database Migrations

This project uses [Alembic](https://alembic.sqlalchemy.org/) for database schema migrations.

## Setup

Alembic is already initialized in `migrations/`. The configuration uses async PostgreSQL (`postgresql+asyncpg`).

## Quick Commands

### Create a new migration

```bash
# After modifying SQLAlchemy models, auto-generate a migration:
alembic revision --autogenerate -m "description of change"

# Or create an empty migration to write manually:
alembic revision -m "description of change"
```

### Apply migrations

```bash
# Run all pending migrations:
alembic upgrade head

# Run one migration:
alembic upgrade +1

# Run to a specific revision:
alembic upgrade 002_hafidz_ledger
```

### Rollback migrations

```bash
# Rollback one migration:
alembic downgrade -1

# Rollback all migrations:
alembic downgrade base
```

### Stamp existing database

For production databases that already have the schema (created before Alembic was introduced):

```bash
# Mark the database as having the baseline migration applied:
alembic stamp 001_baseline

# Then future migrations can be applied normally:
alembic upgrade head
```

### View current state

```bash
# Current revision:
alembic current

# History of all migrations:
alembic history --verbose
```

## Migration Naming Convention

| Prefix | Meaning | Example |
|---|---|---|
| `feat` | New table/column/feature | `feat: add hafidz_contributions table` |
| `fix` | Fix schema issue | `fix: add missing index on messages` |
| `refactor` | Rename/drop without data loss | `refactor: rename experiments.status values` |

## Important Rules

1. **Never edit a committed migration.** Create a new migration to fix issues.
2. **Always test downgrade.** Every migration must have a working `downgrade()`.
3. **Review autogenerated migrations.** Alembic may miss indexes, constraints, or RLS policies.
4. **Migrations must be deterministic.** Same migration = same result on any database.
5. **Keep migrations fast.** Avoid data migrations in the same transaction as schema changes for large tables.

## Troubleshooting

### "Can't locate revision"

Your database is ahead of the migration files. Run:
```bash
alembic stamp head
```

### "Target database is not up to date"

There are pending migrations. Run:
```bash
alembic upgrade head
```

### Async driver issues

Alembic uses the sync wrapper internally. The `env.py` handles async-to-sync translation automatically.
